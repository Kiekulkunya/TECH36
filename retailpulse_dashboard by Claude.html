Dataset Origin
This dashboard is powered by the Retail Intelligence 100K User Behavior Dataset — a rich synthetic dataset simulating real-world e-commerce activity across 108,584 events. It captures the full user journey from product views through to purchase or abandonment, across 9 countries, 8 product categories, and 7 traffic acquisition channels.

Dataset Specifications
  • 108,584 rows — one row per user event (view, click, wishlist, add_to_cart, purchase, drop)
  • 18 columns — identifiers, behavioral, product, context, and outcome fields
  • 9 regions — US, UK, JP, IN, AU, CA, BR, DE, FR with near-uniform distribution
  • 8 categories — Electronics, Groceries, Sports, Accessories, Beauty, Home, Shoes, Apparel
  • 33 brands — led by Nike (11,275 events), Adidas (9,250), Puma (7,036)
  • 3.87% conversion rate — 4,203 purchase events out of 108,584 total
  • 12.70% drop-off rate — 13,797 abandoned sessions
  • Jan 1 – May 1 2026 date range, timestamped in UTC
Engineered Feature
The clean file adds a channel_device_mismatch binary flag identifying 8,938 rows (8.23%) where the reported channel and device type are logically inconsistent — for example, a "web" channel event attributed to an "android" device label. This feature was engineered for data quality analysis.

Build Summary
  • Built a dark-themed D3.js dashboard with 4 real charts: trapezoid funnel, heatmap, stacked bar, and hourly line chart.
  • Added cross-chart interactivity: clicking a heatmap region filters all 4 charts simultaneously with pill badges.
  • Clicking any stacked bar segment (e.g. social + ios) drills the funnel to that traffic/device combo.
  • Hovering a funnel stage loads a live product table showing top 5 products at that action stage.
  • Added light/dark theme toggle using CSS custom properties, persisted to localStorage with system preference fallback.
  • Built CSV drag-and-drop upload with PapaParse, column validation, progress bar, and live chart recomputation.
  • Added an AI Analyst sidebar powered by Gemini 1.5 Flash with token-by-token SSE streaming and simulated fallback.
  • Context injection dynamically builds a rich analyst prompt from active filters, KPIs, and funnel state on every question.
  • Fixed a critical "cannot use 'in' operator" bug from a broken category filter fallback referencing an undefined object.
  • Added this Data & Deep Dive tab with 4 sub-tabs: source, takeaways, reflection, and a review submission form.
Short Reflection
~500 words · What stood out, what surprised me, what I'd do differently
Claude's Reflection

What started as a fairly standard "build me a dashboard" prompt turned into something I genuinely found engaging to think through. The dataset itself is interesting — it's synthetic, but it's synthetic in a deliberate way. The near-perfect balance across regions, categories, and traffic sources is almost too clean, and that cleanliness actually makes it a better teaching dataset. You can see the structure without fighting noise.

The funnel chart was the most satisfying piece to get right. A bar chart would have been easier — just rectangles. But a true trapezoid funnel, where each stage narrows into the next using D3 path commands, actually communicates something that a bar chart can't: the visual narrowing IS the data. When you hover over the click→wishlist gap and see "▼ 74.9% drop", that's not just a number — it's a shape.

The cross-chart interactivity surprised me in how much complexity hides behind "clicking a region updates all charts." You need to pre-aggregate data at multiple levels (all data, per-region, per-segment, per-category), keep that state consistent across 4 different draw functions, manage spinners, and make sure resetting one filter doesn't break another. The bug we hit — the cannot use 'in' operator error — was a good reminder that lazy shorthand like (c in window.CAT_DATA||{}) reads plausibly but evaluates differently than intended. The fix was just cleaner code.

The Gemini integration was the most uncertain piece going in. Streaming SSE responses from a generative model into a chat UI sounds complex, but the Gemini API's alt=sse endpoint is actually well-structured. The interesting design challenge was the context injection — how do you summarize "the current state of a live filtered dashboard" into a prompt that gives the model enough to answer specifically? The answer ended up being: just serialize it all. Region filter, segment filter, KPI values, funnel counts, peak hour. The model doesn't need elegant summarization — it needs the numbers.

If I were continuing this project, I'd add real brush selection on the line chart (drag a time window, see the funnel update), a scatter plot of session_length vs. conversion to find the "engagement threshold," and proper unit tests for the aggregation functions. The CSV upload is functional but fragile — it makes assumptions about column names that a real production tool wouldn't. And the simulated AI responses are charming but obviously limited.

What I found most interesting overall: the 3.87% conversion rate looks bad in isolation, but when you drill JP + Electronics, you get 5.09% — 32% above average. That kind of segment-level insight is invisible in aggregate stats and only visible when the filters work correctly. That's the whole point of the tool.
