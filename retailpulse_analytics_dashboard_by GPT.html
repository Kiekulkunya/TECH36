Data Source

This dashboard uses the retail behavior dataset from Kaggle:

Retail Intelligence 100K User Behavior Dataset — retail_user_behavior_100k.csv

The dataset tracks event-level retail user behavior, including session IDs, user actions, product categories, region, traffic source, device type, conversion flag, drop-off flag, price, and engagement metrics.

  • Added KPI cards for sessions, users, conversion, drop-off, and average price.
  • Created a D3 horizontal funnel for view, click, wishlist, cart, purchase, and drop.
  • Built a region-by-category conversion heatmap with tooltip details.
  • Added a stacked bar chart for traffic source and device mix.
  • Added an hourly line chart for events, conversions, and drop-offs.

Short reflection

What I found most interesting was how a simple event-level retail dataset can turn into a full analytical product. At first, the data looked like normal user behavior logs, but once we connected actions, categories, regions, devices, and traffic sources, it became a story about how customers move through a retail journey.

I learned that dashboard design is not only about making charts. It is about choosing the right sequence of questions: who visits, what they do, where they drop, and which segments convert better. The funnel chart made the customer journey easy to understand, while the heatmap and stacked bar chart helped show performance differences across regions, categories, traffic sources, and devices.

One surprise was how important data structure is for interactivity. The dashboard needed aggregate data instead of loading the full raw CSV by default, but cross-filtering required careful design so the charts would not break. The earlier desktop bug showed that even one missing aggregate object can break the whole experience, so defensive coding and console debugging were very important.

I also found the AI analyst sidebar interesting because it turns a static dashboard into something more conversational. Instead of only reading charts, users can ask direct questions about the current filter context. A question I still have is how far this could go if the dashboard used real-time data and stronger backend analytics instead of only browser-side aggregation.
